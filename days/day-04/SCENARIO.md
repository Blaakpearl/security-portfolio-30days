## 🎯 Threat Brief

It's 02:17 on a Thursday morning. Your EDR platform fires a low-priority behavioral
alert — a single Windows workstation, **DESKTOP-FIN-047**, belonging to a senior
analyst in the Fixed Income trading desk, is generating an unusual volume of
outbound DNS queries. The alert is low confidence, auto-suppressed by a noise
filter, and would normally age out without review.

But you're running a scheduled overnight hunt. You pull the thread.

The DNS queries themselves look normal at first glance — short labels, common
TLD structure, resolving to external IPs. But the **timing is wrong**. Every
60.3 seconds, without variance, the host fires a DNS query to the same external
resolver. 340 times in the past 6 hours. While the trading desk analyst is home
asleep.

You're looking at a **Command & Control beacon**.

The question isn't whether the host is compromised — the regularity of that
interval makes the answer clear. The question is: **how long has this been
running, how far has it spread, and what has the attacker already done?**

## 🔍 The Beacon Hunter's Framework

C2 beacon detection requires moving beyond single-packet inspection to
**statistical pattern analysis** across time. The key insight: human network
traffic is inherently irregular — web browsing, email, application use all
generate bursts with natural variation. Malware beacons are the opposite —
mathematically regular, generated by `sleep()` timers in code.

```
┌──────────────────────────────────────────────────────────────────┐
│  DETECTION METHODOLOGY                                            │
│                                                                   │
│  Step 1: IDENTIFY the beacon                                      │
│    • Frequency analysis — consistent inter-packet timing         │
│    • Long-connection analysis — persistent low-data sessions     │
│    • DNS query regularity — same domain, fixed interval          │
│                                                                   │
│  Step 2: CHARACTERIZE the protocol                                │
│    • JA3/JA3S fingerprinting — TLS client hello analysis         │
│    • DNS query type distribution (A vs TXT vs CNAME)             │
│    • Payload size patterns — data-over-DNS detection             │
│                                                                   │
│  Step 3: PIVOT from the beacon                                    │
│    • Same C2 IP — other internal hosts beaconing?                │
│    • Same JA3 hash — same malware family elsewhere?              │
│    • DNS resolver — is DoH being abused to bypass inspection?    │
│                                                                   │
│  Step 4: SCOPE the compromise                                     │
│    • Lateral movement from beaconing host                        │
│    • Data exfiltration volume via DNS payload analysis           │
│    • Persistence mechanisms on affected host                     │
└──────────────────────────────────────────────────────────────────┘
```

## ⚠️ Detection Challenge

DNS-based C2 is one of the most difficult channels to detect and block:

- **DNS is ubiquitous** — blocking all DNS breaks the network; you can't simply
  firewall it
- **Encryption** — DNS over HTTPS (DoH) hides query content from network inspection
  tools that only inspect plaintext DNS (UDP/53)
- **Legitimate resolvers** — attackers route C2 through `8.8.8.8` (Google) or
  `1.1.1.1` (Cloudflare) via DoH, making the destination IP appear legitimate
- **Low data volume** — DNS beacons often carry only a few bytes per query;
  data exfiltration via DNS is slow but nearly invisible in standard monitoring
- **Sleep jitter** — sophisticated beacons add ±5–15% random variation to their
  interval to defeat simple timing-based detection; requires statistical analysis
  not threshold rules

## 📚 Learning Objectives

1. Perform frequency analysis on DNS query logs to identify C2 beaconing patterns
2. Use Zeek (Bro) to parse PCAP files and extract DNS, HTTP, and connection metadata
3. Apply JA3/JA3S TLS fingerprinting to identify malware families by TLS behavior
4. Detect DNS tunneling by analyzing query entropy, length, and TXT record frequency
5. Write statistical beacon detection logic in Python using inter-arrival time analysis
6. Produce Sigma detection rules for both beacon timing and DNS tunnel indicators
7. Build a network forensics timeline linking the beacon to other host activity

## ✅ Success Criteria

- [ ] Confirm beaconing via inter-arrival time (IAT) statistical analysis — mean, std dev, coefficient of variation
- [ ] Extract the C2 IP, domain, JA3 hash, and beacon interval from PCAP/logs
- [ ] Detect DNS tunneling indicators — query length distribution, TXT record ratio
- [ ] Identify whether any other internal hosts are communicating with the same C2
- [ ] Produce a Zeek script that outputs a beaconing score for all connections
- [ ] Write two Sigma rules: beacon timing + DNS tunnel
- [ ] Build a 5-event network forensic timeline for the compromise
